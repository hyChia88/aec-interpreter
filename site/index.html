AEC Interpreter: Cross-Modal Grounding & Ontology-Driven Retrieval A neuro-symbolic interpreter that maps unstructured site evidence to the exact IFC element in a BIM knowledge graph.

Construction-site evidence is egocentric and messy: phone photos, marked plans, and short notes. BIM is allocentric and structured: IFC types, topology, and globally unique element IDs. AEC Interpreter is the translation layer between them. It answers the first question every downstream workflow depends on: where exactly is this in the model?, using only the BIM model itself, with no history of labelled site photos, and it refuses to guess when it is unsure.

The ProblemThe physical-to-digital traceability gap

Construction and facilities teams standardise on OpenBIM/IFC, yet day-to-day coordination still runs on manual interpretive labour: people translate unstructured evidence — photos, chat messages, floorplan markups — into model-linked records by hand. The front-line worker sees a local condition but not the BIM GUID; the coordinator sees the model but not the site context. In dense buildings, that gap becomes a grounding bottleneck: many windows, doors, and walls are visually and semantically repeated.

The final step is choosing one element from a retrieved shortlist that already contains the answer (median 76 candidates, ground truth present 100% of the time). So the difficulty is not simply retrieval but discriminating among visually identical siblings. What separates two windows is not in their pixels; it is in the building's relational structure. A black-box matcher plateaus at 6.7% right-first with the answer already in the pool. The contribution is to make that relational signal explicit as a spatial address that can be read from evidence and executed against BIM.

WorkflowEvidence first, graph reasoning second, 3D verification last

Neuro-symbolic means a neural network does the perceiving and a deterministic symbolic engine does the deciding. In this project, ontology is the guardrail — legal IFC vocabulary, storeys, classes, and GUIDs — while topology is the discriminator: the local spatial structure that separates look-alike elements. Evidence flows through one pipeline:

The load-bearing design choice is auditable execution over free-form generation: the neural network never writes a database query. It emits a structured description; deterministic templates turn that into graph operations. The neural side translates evidence into a typed spatial address; the symbolic side names only GUIDs that exist in the model. Every error is therefore traceable to a specific field or stage — type, floor, position, confidence, or ranking.

EngineeringEngineering decisions, measured against alternatives

The project was built as a set of explicit contracts rather than one opaque model call. Each contract has an experiment behind it: what to let the VLM do, what to push into deterministic geometry, when to trust an extracted address, and how much graph context is recoverable from site evidence. This is the ownership layer: IFC parsing, graph enrichment, constrained generation, calibrated routing, evaluation harnesses, figures, and the browser demo all expose where the system succeeds or fails.

Should the VLM directly name the BIM element? No. The VLM emits typed JSON; deterministic graph queries name only IFC GUIDs that exist in the model. Should noisy address fields hard-filter candidates? No. The address is a soft prior with calibrated answer/defer routing, preserving recall when the detector is uncertain. Evidence flowThe engineered pieces, in execution order

② Perception — reading the evidenceFrom pixels and prose to a typed description

Fine-tuned VLM — the perception engine. A LoRA-tuned Qwen2.5-VL reads photo, note, and floorplan evidence into a typed contract: {storey, ifc_class, spatial_relations[]}. It describes; it never queries the graph. Direct impact: spatial-direction accuracy improves from 0% zero-shot Gemini to 82% after fine-tuning.  Qwen2.5-VL · LoRA · Constraints JSON
Deterministic visual specialists. OpenCV counts openings and reads ordinal position; ResNet classifies size bands — the sub-tasks a VLM does unreliably. Each emits its own confidence, and this is the seed of the spatial-interpretation work below.  OpenCV · ResNet

③ Symbolic backend — the deterministic engineAn IFC-native knowledge graph + a deterministic planner

IFC Parse & Enrich Engine. Converts the raw IFC into a knowledge graph — nodes are elements, edges are relations — enriched beyond native containment with retrieval-oriented topology: NEXT_TO, CONNECTS_TO, ADJACENT_TO, ON_STOREY.  IfcOpenShell · Neo4j
Symbolic Planner + Graph-RAG. A recall-preserving p0∪p1 union planner combines attribute filtering with spatial-topology ranking; templates are deterministic — same input, same query, same result.  Cypher · priority cascade

④ Control — adaptive routingAn orchestration layer that knows when to defer

Orchestration & routing

Confidence-driven control

Every extracted field carries a {value, confidence, source} record. An orchestration layer routes on that confidence — which deterministic specialist to trust, which constraint to apply softly, and when to defer to a human instead of guessing. The policy is studied as an ablation, from a static calibrated threshold → a learned router → an LLM-agent orchestrator (same backbone), scored on accuracy, latency, and repeatability.

static → learned router → agent · the adaptivity arm
Trained with zero real labels. Skeleton-first generation fixes ground truth before adding stochastic visual evidence: IFC mining → Blender renders with hard negatives → LLM text → LLM-as-judge filter → 990 cases, no human annotation.  Blender · Gemini · LLM-judge

InnovationReading space from a flat image — the spatial-interpretation layer

The thesis proved the architecture sound: with perfect perception the symbolic engine keeps the correct element 100% of the time and compresses the shortlist toward a single answer — so the bottleneck is perception, not graph logic. The core advance deepens the deterministic visual specialists into a principled spatial-interpretation layer: instead of ad-hoc cues, the system recovers a spatial address — a relational key read from plain 2D images and floorplans, computable from the BIM model and recoverable from the evidence. Three questions, answered with measurements:

① Representation

What is the minimal address?

It is type-conditional: a coarse prefix (storey + class) is necessary but saturated; the discriminator is class-specific — a position-slot (i of M) for an opening, a connectivity fingerprint for a wall.

ceiling: right-first 4.9% → 78.5% (n=60)
② Mechanism

How to use a noisy address?

Hard filtering deletes the answer, so the address is a soft prior in a recall-fixed pool. One real detector closes most of the gap; its confidence passes a calibration gate, and its payoff is knowing when to abstain — not blind reweighting.

realized slot: 6.6% → 67.6% · defer → 80.6% (n=35)
③ Architecture

How deep should context go?

A depth law: deeper relations are more unique but their recovery from an image collapses with distance, so discrimination saturates at one hop — compile the relation into the element rather than chase deep chains.

confusable set 13 → 8 at one hop

ExamplesThe evidence path, simplified

The demo is intentionally inspectable. A case moves from raw site evidence to field extraction, then to a spatial-address knowledge graph, then to candidate ranking and a 3D BIM highlight. The key is not one monolithic answer screen; it is seeing which evidence field and which graph relation caused the final GUID decision. The full benchmark uses 60 held-out AP cases; the live static viewer shows 6 representative held-out cases so the interface stays fast.

Raw evidence. A site photo and note start the grounding task.
Field extraction. The VLM handles coarse semantics; visual specialists handle sibling-level address fields.
Spatial address. The extracted slot or fingerprint becomes an auditable graph record, not free text.
Confidence routing. Low-confidence address fields route to defer instead of a confident wrong GUID.
Answer case. Predicted "slot 2 of 17" is correct and above threshold.
3D verification. The selected IFC element is highlighted in the BIM viewer for coordinator review.

Answer vs defer

Each detailed card traces one case top-to-bottom: raw input → expected address → detector reading → predicted address + decision. The predicted address is drawn as a graph, mirroring the ground-truth graph above it.

ANSWER  Predicted "slot 2 of 17" — ✓ correct; confidence above threshold, so the system commits to the GUID.
DEFER  Predicted "1 of 10" but truth is "8 of 10" (wrong); confidence 0.05 is below threshold, so the system abstains and returns candidates rather than a confident mistake — the triage value proposition.
